# P6.4 — DevOps Automation Agent

**Phase:** 6 — AI Agents  
**Topic:** Agents, DevOps, Human-in-the-Loop  
**Type:** Project

---

## Goal
Build an agent with access to shell commands, GitHub API, and Slack, that can diagnose failing CI builds, analyze logs, and suggest or apply fixes — with human approval before any destructive action.

---

## What to Build

### 1. Tool Suite
```python
tools = [
    # Read-only (auto-approved)
    run_shell_readonly(cmd: str) -> str,   # only allows: ls, cat, grep, ps, df, top, git log, git diff, git status
    read_file(path: str) -> str,
    get_github_pr(repo, pr_number) -> dict,
    get_github_run_logs(repo, run_id) -> str,
    search_logs(pattern: str, file: str) -> List[str],
    
    # Write (requires approval)
    write_file(path: str, content: str),
    run_shell_write(cmd: str) -> str,       # git add, commit, push, etc.
    post_github_comment(repo, pr, message),
    send_slack_message(channel, message),
    trigger_github_action(repo, workflow),
]
```

### 2. Use Case 1 — CI Build Diagnostics
The agent receives: "The CI build failed for PR #42 in repo owner/repo. Diagnose it."

Agent workflow:
1. Fetch the failed GitHub Actions run logs
2. Identify the failing step and error message
3. Fetch the PR diff
4. Cross-reference: is the error related to the code changes?
5. Search existing issues/PRs for similar failures
6. Output: structured diagnosis with root cause + fix suggestion

```markdown
## Build Failure Diagnosis — PR #42

**Failing Step**: Run Tests
**Error**: `ModuleNotFoundError: No module named 'pydantic_v2'`

**Root Cause**: The PR added `pydantic>=2.0` to requirements but the CI environment 
uses the old pydantic v1 API. The model validators use `@validator` (v1 syntax).

**Suggested Fix**:
- Option A: Pin `pydantic<2.0` in requirements.txt (quick fix)
- Option B: Migrate validators to `@field_validator` (pydantic v2 syntax)

**Confidence**: High (same error seen in PR #38)

Would you like me to apply Option A? [y/n]
```

### 3. Use Case 2 — Log Analysis
Agent receives: "Our API has been returning 500 errors for the last 2 hours. Check the logs."

Agent workflow:
1. Read the application log file
2. Extract all error lines + stack traces
3. Group similar errors (by exception type + location)
4. Identify the most frequent error
5. Correlate with recent deployments (check git log)
6. Output diagnosis + propose fix

### 4. Use Case 3 — Automated Fix (with Approval)
When the agent suggests a fix and user approves:
1. Apply the code change
2. Run tests locally
3. If tests pass: commit with message, push to a new branch, open a draft PR
4. Post a Slack message: "Fix applied and PR opened: #43"

Approval gates:
- Before writing any file: show diff, ask for approval
- Before git push: show commit message and target branch, ask for approval
- Before Slack message: show message content, ask for approval

### 5. Slack Bot Integration
- Connect to Slack via webhook or Slack SDK
- Users can trigger the agent via Slack: `@devops-bot diagnose PR #42`
- Agent responds in Slack with the diagnosis
- User approves fixes directly in Slack with button clicks
