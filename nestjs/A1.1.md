**Phase:** 1 — NestJS Fundamentals  
**Topic:** Modules, controllers, providers, DI container, decorators, metadata  
**Type:** Assignment

---

## Goal
Understand how NestJS assembles an application from decorated TypeScript classes and how the IoC container wires dependencies automatically.

## Background

NestJS is built on top of Express (or Fastify) and adds a structured, opinionated layer inspired by Angular. The core of this structure is the **module system** and **dependency injection (DI) container**.

Every NestJS application has exactly one root module (`AppModule`) and any number of feature modules. At startup, NestJS reads the `@Module()` metadata attached to each module class and builds a directed acyclic graph of providers. It then resolves all constructor dependencies and instantiates each provider — usually as a singleton — before handling any request.

**Decorators** in NestJS are plain TypeScript decorators that call `Reflect.defineMetadata` under the hood (via the `reflect-metadata` package). `@Module()` stores module metadata, `@Controller()` marks a class as a route handler, `@Injectable()` marks a class as a provider that can be injected. When you write `@Get('/hello')`, NestJS stores that route path and HTTP method as metadata on the handler method, then reads it at bootstrap time to register Express routes.

The **three building blocks** you need to memorise:
- **Module** — organises a cohesive feature slice. Declares which providers it owns (`providers`), which controllers it exposes (`controllers`), what it exports for other modules to use (`exports`), and which other modules it depends on (`imports`).
- **Controller** — receives HTTP requests and delegates to services. Should contain zero business logic.
- **Provider** — any class decorated with `@Injectable()`. Usually a service, repository, guard, pipe, or interceptor. The DI container manages its lifetime.

## Tasks

### 1. Scaffold a new NestJS project

```bash
npm i -g @nestjs/cli
nest new nestjs-fundamentals
cd nestjs-fundamentals
npm run start:dev
```

Visit `http://localhost:3000` — you should see `Hello World!`.

### 2. Inspect the default files

Open `src/app.module.ts`:
```typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';

@Module({
  imports: [],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Open `src/app.service.ts`:
```typescript
import { Injectable } from '@nestjs/common';

@Injectable()
export class AppService {
  getHello(): string {
    return 'Hello World!';
  }
}
```

Open `src/app.controller.ts`:
```typescript
import { Controller, Get } from '@nestjs/common';
import { AppService } from './app.service';

@Controller()
export class AppController {
  constructor(private readonly appService: AppService) {}

  @Get()
  getHello(): string {
    return this.appService.getHello();
  }
}
```

Trace the flow: `AppModule` declares both `AppService` and `AppController`. NestJS sees that `AppController` constructor requires `AppService`, instantiates `AppService` first, then injects it.

### 3. Inspect the metadata yourself

Add a temporary `main.ts` experiment:
```typescript
import 'reflect-metadata';
import { Controller, Get } from '@nestjs/common';

@Controller('demo')
class DemoController {
  @Get('hello')
  sayHello() {}
}

const path = Reflect.getMetadata('path', DemoController);
const routes = Reflect.getMetadata('__routeArguments__', DemoController) ?? 'N/A';
console.log('Controller path:', path);   // 'demo'
```

This shows that decorators are just metadata writers — NestJS reads them at runtime.

### 4. Create a feature module manually

```bash
nest generate module cats
nest generate controller cats --no-spec
nest generate service cats --no-spec
```

This creates:
```
src/cats/
  cats.module.ts
  cats.controller.ts
  cats.service.ts
```

**`src/cats/cats.service.ts`**
```typescript
import { Injectable } from '@nestjs/common';

export interface Cat {
  id: number;
  name: string;
  breed: string;
}

@Injectable()
export class CatsService {
  private cats: Cat[] = [
    { id: 1, name: 'Whiskers', breed: 'Persian' },
  ];

  findAll(): Cat[] {
    return this.cats;
  }
}
```

**`src/cats/cats.controller.ts`**
```typescript
import { Controller, Get } from '@nestjs/common';
import { CatsService, Cat } from './cats.service';

@Controller('cats')
export class CatsController {
  constructor(private readonly catsService: CatsService) {}

  @Get()
  findAll(): Cat[] {
    return this.catsService.findAll();
  }
}
```

**`src/cats/cats.module.ts`**
```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {}
```

**`src/app.module.ts`** — register CatsModule:
```typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Test: `curl http://localhost:3000/cats` — returns the cat array.

### 5. Understand what breaks DI

Temporarily remove `CatsService` from the `providers` array in `CatsModule` and restart. NestJS will throw:

```
Nest can't resolve dependencies of the CatsController (?).
Please make sure that the argument CatsService at index [0]
is available in the CatsModule context.
```

Read this error carefully — it tells you exactly which argument, at which constructor index, in which module context is missing. Restore the providers array.

### 6. Enable module introspection

In `main.ts`, after `app = await NestFactory.create(AppModule)`, log:
```typescript
const moduleRef = app.get(AppModule);
console.log('App running with module:', moduleRef.constructor.name);
```

This confirms the DI container has resolved AppModule.
