## Goal

## Task

### Part 1 — ConfigMaps
```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_HOST: postgres-service
  DATABASE_PORT: "5432"
  LOG_LEVEL: info
  # Multi-line values (config files)
  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://api:8080;
      }
    }
```

Use a ConfigMap in a Pod:
```yaml
spec:
  containers:
    - name: app
      image: myapp:latest
      # Method 1: inject all keys as environment variables
      envFrom:
        - configMapRef:
            name: app-config
      # Method 2: inject specific key
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
      # Method 3: mount as a file
      volumeMounts:
        - name: config-volume
          mountPath: /etc/nginx/conf.d
  volumes:
    - name: config-volume
      configMap:
        name: app-config
        items:
          - key: nginx.conf
            path: default.conf
```

```bash
kubectl apply -f configmap.yaml
kubectl get configmap app-config -o yaml
kubectl describe configmap app-config

# Update a ConfigMap (pods need restart to pick up changes unless mounted as volume)
kubectl edit configmap app-config
# or
kubectl patch configmap app-config --patch '{"data":{"LOG_LEVEL":"debug"}}'
```

### Part 2 — Secrets
```yaml
# Secrets are base64 encoded (NOT encrypted by default!)
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: cG9zdGdyZXM=       # echo -n "postgres" | base64
  password: c2VjcmV0cGFzcw==   # echo -n "secretpass" | base64
```

```bash
# Create a Secret from literal values (easier)
kubectl create secret generic db-credentials \
    --from-literal=username=postgres \
    --from-literal=password=secretpass

# Create from a file (good for TLS certs)
kubectl create secret generic tls-certs \
    --from-file=tls.crt=./server.crt \
    --from-file=tls.key=./server.key

# Create TLS secret (special type)
kubectl create secret tls myapp-tls \
    --cert=server.crt \
    --key=server.key
```

Use secrets:
```yaml
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: password
```

**Important**: By default, Kubernetes Secrets are stored unencrypted in etcd. Learn about Sealed Secrets or External Secrets Operator to fix this:
```bash
# Sealed Secrets: encrypt with a cluster-specific key
kubeseal --format yaml < db-credentials.yaml > db-credentials-sealed.yaml
# Now you can safely commit db-credentials-sealed.yaml to git
```

### Part 3 — PersistentVolumes and PersistentVolumeClaims
```yaml
# PersistentVolume (cluster admin creates these — or cloud provider does automatically)
apiVersion: v1
kind: PersistentVolume
metadata:
  name: local-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: local-storage
  hostPath:
    path: /data/pv  # on the node (not recommended for production)

---
# PersistentVolumeClaim (developer requests storage)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard  # on minikube; use gp2 on AWS EKS
```

```yaml
# Use PVC in a Deployment
spec:
  template:
    spec:
      containers:
        - name: postgres
          image: postgres:16
          volumeMounts:
            - name: postgres-storage
              mountPath: /var/lib/postgresql/data
      volumes:
        - name: postgres-storage
          persistentVolumeClaim:
            claimName: postgres-pvc
```

```bash
kubectl get pv
kubectl get pvc
kubectl describe pvc postgres-pvc  # check it's "Bound"
```

### Part 4 — StorageClasses
```yaml
# StorageClass defines how storage is provisioned dynamically
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Delete  # or Retain
```

On cloud providers:
```yaml
# AWS EKS
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"

# GKE
provisioner: pd.csi.storage.gke.io
parameters:
  type: pd-ssd
```

### Part 5 — StatefulSets (for Databases)
```yaml
# StatefulSet gives stable network identity and ordered deployment
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: db-credentials
                  key: password
          volumeMounts:
            - name: postgres-data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:   # PVC created per replica automatically
    - metadata:
        name: postgres-data
      spec:
        accessModes: [ReadWriteOnce]
        resources:
          requests:
            storage: 5Gi
```

Explain: when would you use a StatefulSet instead of a Deployment?
