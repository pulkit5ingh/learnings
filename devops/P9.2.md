## SLO Framework

### Define SLOs for All Services
```yaml
# slos/services.yaml — The source of truth for all SLOs
services:
  api:
    availability:
      target: 99.5
      measurement_window: 30d
      owner: backend-team
    latency_p99:
      target_ms: 1000
      measurement_window: 5m
    error_budget_alert_at: 50  # alert when 50% consumed

  database:
    availability:
      target: 99.9
      measurement_window: 30d
      owner: platform-team
    query_p95:
      target_ms: 100

  frontend:
    availability:
      target: 99.0
    page_load_p75:
      target_ms: 3000  # Core Web Vital threshold
```

### Prometheus Recording Rules (Generated from slos.yaml)
```python
# scripts/generate-rules.py — generate Prometheus rules from SLOs
import yaml, jinja2

SLO_TEMPLATE = """
groups:
  - name: slo_{{ service }}
    rules:
      - record: job:http_availability:rate5m
        expr: |
          sum(rate(http_requests_total{service="{{ service }}",status!~"5.."}[5m]))
          / sum(rate(http_requests_total{service="{{ service }}"}[5m]))

      - alert: {{ service|title }}ErrorBudgetBurn
        expr: |
          (1 - job:http_availability:rate5m{service="{{ service }}"}) 
          / {{ (1 - slo.availability.target/100) | round(4) }} > 14.4
        for: 2m
        labels:
          severity: critical
          service: {{ service }}
          slo: availability
        annotations:
          summary: "{{ service }} burning error budget 14.4x (exhaustion in 2h)"
"""

with open('slos/services.yaml') as f:
    slos = yaml.safe_load(f)

for service, config in slos['services'].items():
    rules = jinja2.Template(SLO_TEMPLATE).render(service=service, slo=config)
    with open(f'prometheus/rules/slo-{service}.yaml', 'w') as f:
        f.write(rules)
```

## Grafana SLO Dashboard

### Panel Structure
```
Row 1: SLO Status (all services)
  - Current availability vs target (color: green/yellow/red)
  - Error budget remaining (days until exhaustion)
  
Row 2: Error Budget Detail (selected service)
  - 30-day rolling availability
  - Error budget consumption rate
  - Projected exhaustion date
  
Row 3: Traffic and Errors
  - Request rate
  - Error rate by endpoint
  
Row 4: Latency
  - P50/P95/P99 over time
  - Slow endpoints table
```

### Error Budget Remaining Panel
```promql
# Days of error budget remaining
(
  (1 - avg_over_time(job:http_availability:rate5m[30d])) - (1 - 0.995)
) / (1 - 0.995) * 30
```

### Projected Exhaustion
```promql
# If we keep burning at the current rate, when do we run out?
30 * (0.005 - (1 - avg_over_time(job:http_availability:rate5m[30d]))) 
/ max_over_time(job:http_error_rate:rate1h[24h])
```

## Quick Diagnosis (< 2 minutes)

```bash
# 1. Current error rate
kubectl exec -n monitoring prometheus-0 -- \
    promtool query instant 'rate(http_requests_total{status=~"5.."}[5m])'

# 2. Recent deployments
kubectl rollout history deployment -n production

# 3. Top error messages
kubectl logs -l app=api -n production --since=5m | grep ERROR | sort | uniq -c | sort -rn | head 20

# 4. Database connectivity
kubectl exec -n production deploy/api -- python -c "from app.db import engine; engine.connect()"
```

## Decision Tree

```
Error rate > 1%?
│
├─ Yes: Recent deployment (< 30 min)?
│       ├─ Yes → ROLLBACK: kubectl rollout undo deploy/api -n production
│       └─ No → Check logs for pattern
│               ├─ DB connection errors → See database runbooks
│               ├─ Timeout errors → Check downstream services
│               └─ 404 errors → Check routing/ingress
│
└─ No: False alarm? Check alert definition
```

## Common Causes and Resolutions

| Cause | Signals | Resolution |
|-------|---------|------------|
| Bad deployment | Errors started at deployment time | Rollback |
| DB overloaded | High DB query time | Scale DB or add read replica |
| Memory leak | Pod OOMKilled repeatedly | Restart pods, fix root cause |
| Dependency down | Errors on one endpoint | Enable fallback/circuit breaker |
| Config change | Errors started after config update | Revert config |

## Escalation Matrix
| Severity | What | Action |
|----------|------|--------|
| P1 | SLA breach | Wake everyone up. Now. |
| P2 | SLO breach, users affected | Escalate after 15 min unresolved |
| P3 | Degraded performance | Fix during business hours |
| P4 | Warning, no user impact | Ticket, fix in sprint |

## Handoff Checklist
- [ ] Any active incidents? Status?
- [ ] Any SLOs at risk?
- [ ] Any upcoming deployments?
- [ ] Any known fragile parts of the system right now?

**Setup**: Set connection pool max to 5
**Expected behavior**: 503 errors with retry logic, not 500 crashes

**Setup**: Make order-service call user-service with 5s timeout

**Expected behavior**: order-service degrades gracefully (fallback)
**Learning**: Why circuit breakers matter
