# A3.4 — Container Image Security Scanning with Trivy

**Phase:** 3 — Containers (Docker)  
**Topic:** Container security, vulnerability scanning  
**Type:** Assignment

---

## Goal
Scan container images for vulnerabilities, understand CVE severity levels, and implement security best practices that reduce your attack surface.

---

## Task

### Part 1 — Install and Run Trivy
```bash
# Install Trivy
brew install aquasecurity/trivy/trivy  # macOS
# or
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh

# Scan an image
trivy image python:3.11        # full scan (will find many vulns in full image)
trivy image python:3.11-slim   # slim has far fewer
trivy image python:3.11-alpine # alpine has the fewest

# Compare vulnerability counts between base images
```

Tip: run all 3 and put the counts in a table.

### Part 2 — Understanding CVE Output
```bash
# Scan with detailed output
trivy image --severity HIGH,CRITICAL python:3.11-slim

# Output in different formats
trivy image --format json --output vulns.json python:3.11-slim
trivy image --format sarif --output vulns.sarif python:3.11-slim  # GitHub format
trivy image --format table python:3.11-slim
```

Read the output and understand each field:
- `LIBRARY` — which package has the vulnerability
- `VULNERABILITY ID` — CVE identifier (e.g., CVE-2023-12345)
- `SEVERITY` — CRITICAL, HIGH, MEDIUM, LOW, UNKNOWN
- `INSTALLED VERSION` — what version is installed
- `FIXED VERSION` — what version fixes it (empty = no fix yet)

### Part 3 — Scan Your Own Dockerfile
Build the Dockerfiles you created in A3.1 and scan them:
```bash
docker build -t myapp-bad -f Dockerfile.bad .
docker build -t myapp-good -f Dockerfile.good .

trivy image --severity HIGH,CRITICAL myapp-bad
trivy image --severity HIGH,CRITICAL myapp-good

# Did your optimized Dockerfile reduce vulnerabilities?
```

### Part 4 — Scan for Misconfigurations (Not Just Vulnerabilities)
```bash
# Scan Dockerfile for bad practices
trivy config Dockerfile

# Common misconfigurations it detects:
# - Running as root
# - Using ADD instead of COPY
# - Pinning to :latest tag
# - Missing USER instruction
# - Missing HEALTHCHECK

# Scan docker-compose.yml
trivy config docker-compose.yml

# Scan Kubernetes manifests
trivy config k8s/
```

### Part 5 — Fix Vulnerabilities
For each HIGH/CRITICAL vulnerability found in your app image:

1. Check if there is a fix available (`FIXED VERSION` column)
2. Update the base image or package to the fixed version
3. Re-scan to confirm it's gone

```dockerfile
# Pin specific package versions in Dockerfile
RUN apt-get update && apt-get install -y \
    libssl3=3.0.11-1 \
    && rm -rf /var/lib/apt/lists/*

# Or upgrade to fixed base image
FROM python:3.11.8-slim  # specific patch version
```

### Part 6 — Integrate Trivy in CI/CD
```yaml
# .github/workflows/security.yml
- name: Scan Docker image with Trivy
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    format: sarif
    output: trivy-results.sarif
    severity: CRITICAL,HIGH
    exit-code: '1'  # fail the pipeline on HIGH/CRITICAL

- name: Upload Trivy results to GitHub Security tab
  uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: trivy-results.sarif
```

### Part 7 — Image Signing with Cosign
```bash
# Install cosign
brew install sigstore/tap/cosign

# Generate key pair
cosign generate-key-pair

# Sign an image (after pushing to registry)
cosign sign --key cosign.key ghcr.io/myorg/myapp:latest

# Verify the signature
cosign verify --key cosign.pub ghcr.io/myorg/myapp:latest

# Verify before pulling (policy enforcement)
# Can be enforced at cluster level with Kyverno or OPA Gatekeeper
```
