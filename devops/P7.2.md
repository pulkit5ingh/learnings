### Grafana Dashboard Panels
Build a single "SLO Status" dashboard:

**Panel 1: Error Budget Gauge**
```promql
# Remaining error budget (%)
(0.005 - (1 - avg_over_time(job:http_availability:rate5m[30d]))) 
/ 0.005 * 100
```

**Panel 2: Burn Rate Heatmap**

## Incident Response Automation

### PagerDuty + Slack Integration
When a critical alert fires:
1. PagerDuty pages on-call engineer
2. Lambda function creates a Slack channel: `#incident-2025-06-01`
3. Runbook link posted automatically in the channel
4. Status page updated (Statuspage.io)

```python
# lambda/incident-create/handler.py
import boto3, json, os
import requests

SLACK_TOKEN = os.environ['SLACK_BOT_TOKEN']

def handler(event, context):
    alert = json.loads(event['body'])
    alert_name = alert['commonLabels']['alertname']
    
    # Create Slack channel
    date_str = datetime.now().strftime('%Y-%m-%d-%H%M')
    channel_name = f"incident-{date_str}-{alert_name.lower().replace('_', '-')}"
    
    channel = requests.post(
        'https://slack.com/api/conversations.create',
        headers={'Authorization': f'Bearer {SLACK_TOKEN}'},
        json={'name': channel_name, 'is_private': False}
    ).json()
    
    # Post runbook link
    requests.post(
        'https://slack.com/api/chat.postMessage',
        headers={'Authorization': f'Bearer {SLACK_TOKEN}'},
        json={
            'channel': channel['channel']['id'],
            'text': f':rotating_light: *Incident: {alert_name}*\n'
                    f'Runbook: https://wiki.company.com/runbooks/{alert_name}\n'
                    f'Dashboard: https://grafana.company.com/d/...'
        }
    )
```

## Postmortem Template
```markdown
# Postmortem: [Incident Title]

**Date**: 2025-06-01  
**Duration**: 47 minutes (14:23 – 15:10 UTC)  
**Severity**: P1 (SLA breach)  
**On-call**: Alice  
**Authors**: Alice, Bob  

## Impact
- 8% of users received 500 errors
- 47 minutes of elevated error rate
- Error budget consumed: 94 of 216 available minutes (43%)

## Timeline (UTC)
| Time  | Event |
|-------|-------|
| 14:20 | Deployment of v2.3.1 began |
| 14:23 | Error rate rose to 8% |
| 14:24 | PagerDuty alert fired |
| 14:27 | Alice acknowledged |
| 14:35 | Root cause identified (N+1 query) |
| 15:07 | Rollback to v2.3.0 complete |
| 15:10 | Error rate returned to baseline |

## Root Cause
PR #234 introduced a user listing endpoint that made one database query per user in the result set (N+1 query pattern). Under normal load (< 50 users per page), latency was acceptable. At peak, with 200 users per page, each request made 200 DB queries, saturating the connection pool.

## Contributing Factors
- No load testing before deployment
- Staging database had 50 users; production has 50,000
- No query count metrics in observability

## What Went Well
- Alert fired within 3 minutes of error rate rising
- Runbook made diagnosis faster (checked recent deployments first)
- Rollback took only 2 minutes

## Action Items
| Action | Owner | Due Date |
|--------|-------|----------|
| Add load test to CI with realistic data | Bob | 2025-06-15 |
| Add db_query_count metric to API | Alice | 2025-06-08 |
| Alert on connection pool exhaustion | Alice | 2025-06-08 |
| Seed staging DB with production-scale data | Bob | 2025-06-20 |

## Lessons Learned
1. Staging data scale matters as much as staging configuration
2. Database queries per request should be a first-class metric
```
