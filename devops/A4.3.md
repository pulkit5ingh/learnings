## Task

### Part 1 — Ingress (HTTP Routing)
Ingress routes external HTTP/HTTPS traffic to internal services based on host and path.

```bash
# Install nginx ingress controller on minikube
minikube addons enable ingress

# Verify
kubectl get pods -n ingress-nginx
```

```yaml
# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/ssl-redirect: "false"
spec:
  ingressClassName: nginx
  rules:
    - host: myapp.local
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-service
                port:
                  number: 8080
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 3000
    - host: admin.myapp.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: admin-service
                port:
                  number: 8081
```

```bash
# Add to /etc/hosts (local testing)
echo "$(minikube ip) myapp.local admin.myapp.local" | sudo tee -a /etc/hosts

kubectl apply -f ingress.yaml
kubectl get ingress
kubectl describe ingress app-ingress

# Test routing
curl http://myapp.local/api/health
curl http://myapp.local/
curl http://admin.myapp.local/
```

### Part 3 — Resource Requests and Limits
```yaml
resources:
  requests:
    memory: "128Mi"   # scheduler uses this to decide which node
    cpu: "100m"       # 100 millicores = 0.1 CPU core
  limits:
    memory: "256Mi"   # container is OOMKilled if it exceeds this
    cpu: "500m"       # container is throttled if it exceeds this
```

**Understanding the difference:**
- `requests`: what the pod needs to START. Used for scheduling.
- `limits`: the maximum it's allowed to use. Enforcement at runtime.

What happens when limits are exceeded:
- CPU: container is throttled (slowed down), not killed
- Memory: container is OOMKilled (terminated)

```bash
# Watch resource usage per pod
kubectl top pods
kubectl top nodes

# See resource requests vs actual usage
kubectl describe node <node-name> | grep -A 5 "Allocated resources"
```

### Part 4 — Horizontal Pod Autoscaler (HPA)
```yaml
# hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-deploy
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70  # scale up when avg CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: AverageValue
          averageValue: 200Mi
```

```bash
# Enable metrics server (required for HPA)
minikube addons enable metrics-server

kubectl apply -f hpa.yaml
kubectl get hpa

# Load test to trigger autoscaling
kubectl run load-generator --image=busybox -- sh -c \
    "while true; do wget -q -O- http://api-service:8080/; done"

# Watch the HPA scale up
kubectl get hpa -w
kubectl get pods -w
```

### Part 5 — QoS Classes and Pod Priority
Kubernetes assigns QoS classes based on resource configuration:

| QoS Class | Condition | Eviction Priority |
|-----------|-----------|-------------------|
| Guaranteed | requests == limits for all containers | Last to be evicted |
| Burstable | requests set but != limits | Middle |
| BestEffort | No requests or limits set | First to be evicted |

```bash
# Check QoS class of a pod
kubectl get pod <name> -o jsonpath='{.status.qosClass}'
```

Best practice: always set `requests = limits` for production workloads (Guaranteed QoS).

### Part 6 — LimitRange and ResourceQuota
```yaml
# LimitRange: default limits for a namespace
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: production
spec:
  limits:
    - default:
        cpu: 500m
        memory: 256Mi
      defaultRequest:
        cpu: 100m
        memory: 128Mi
      type: Container

---
# ResourceQuota: total limits for a namespace
apiVersion: v1
kind: ResourceQuota
metadata:
  name: namespace-quota
  namespace: production
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
    services: "10"
```
