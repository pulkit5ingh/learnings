## Goal
Write production-quality Dockerfiles. A bad Dockerfile creates security vulnerabilities, slow builds, and huge images. This assignment fixes all of that.

---

## Task

### Part 1 — The Naive vs Optimized Dockerfile
Start with this bad Dockerfile for a Python app:
```dockerfile
# BAD — do not use this
FROM python:3.11
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
CMD ["python", "app.py"]
```

Problems:
- Runs as root
- Image is ~1GB (includes build tools)
- Every code change rebuilds dependencies
- No health check
- No .dockerignore

Fix each problem:

```dockerfile
# GOOD — production Dockerfile
FROM python:3.11-slim AS builder

WORKDIR /build
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

FROM python:3.11-slim AS runtime

RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid appgroup --no-create-home appuser

WORKDIR /app
COPY --from=builder /root/.local /home/appuser/.local
COPY --chown=appuser:appgroup src/ ./src/

USER appuser
ENV PATH=/home/appuser/.local/bin:$PATH

HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8080/health')"

EXPOSE 8080
CMD ["python", "-m", "src.main"]
```

Create `.dockerignore`:
```
.git
.env
__pycache__
*.pyc
.pytest_cache
htmlcov
*.egg-info
.venv
node_modules
```

### Part 2 — Multi-Stage Build for a Node.js App
```dockerfile
# Stage 1: Install dependencies
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --only=production

# Stage 2: Build the application
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 3: Production image (minimal)
FROM node:20-alpine AS runner
RUN addgroup --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

WORKDIR /app
ENV NODE_ENV=production

COPY --from=deps --chown=nextjs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nextjs:nodejs /app/dist ./dist
COPY --from=builder --chown=nextjs:nodejs /app/package.json ./

USER nextjs
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

Compare the image sizes: `docker images | grep myapp`

### Part 3 — Layer Caching Strategy
Docker builds layers from top to bottom. Any changed layer invalidates all layers below it.

Experiment:
```bash
# Build 1: initial build (measure time)
time docker build -t myapp:v1 .

# Change only the source code (not requirements.txt)
echo "# comment" >> src/main.py
time docker build -t myapp:v2 .  # should be much faster due to cache

# Change requirements.txt
echo "requests==2.31.0" >> requirements.txt
time docker build -t myapp:v3 .  # dependencies layer cache miss
```

Rule: put things that change least often at the TOP of the Dockerfile.

### Part 4 — Image Size Reduction
```bash
# Check image size and layers
docker image inspect myapp:latest
docker history myapp:latest

# Dive is a tool to explore each layer
docker run --rm -it \
    -v /var/run/docker.sock:/var/run/docker.sock \
    wagoodman/dive myapp:latest
```

Strategies to reduce size:
1. Use `-slim` or `-alpine` base images
2. Combine RUN commands: `RUN apt-get update && apt-get install -y pkg && rm -rf /var/lib/apt/lists/*`
3. Multi-stage builds (copy only what you need)
4. Use `--no-cache-dir` for pip
5. Use `npm ci` instead of `npm install`

Target: get your Python app image under 150MB.

### Part 5 — BuildKit and Build Arguments
```dockerfile
# Build arguments for configuration
ARG PYTHON_VERSION=3.11
ARG APP_ENV=production

FROM python:${PYTHON_VERSION}-slim

ARG APP_ENV
ENV APP_ENV=${APP_ENV}

# Conditional behavior based on build arg
RUN if [ "$APP_ENV" = "development" ]; then \
        pip install debugpy watchdog; \
    fi
```

```bash
# Build with custom args
docker build \
    --build-arg PYTHON_VERSION=3.12 \
    --build-arg APP_ENV=development \
    -t myapp:dev .

# Enable BuildKit (faster, better caching)
DOCKER_BUILDKIT=1 docker build .
# or add to daemon.json: { "features": { "buildkit": true } }
```
