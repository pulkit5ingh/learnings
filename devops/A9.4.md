# A9.4 — Progressive Delivery: Canary Deployment with Argo Rollouts

**Phase:** 9 — Advanced: GitOps, SRE & Platform Engineering  
**Topic:** Progressive delivery, canary, blue-green  
**Type:** Assignment

---

## Goal
Implement progressive delivery — deploy new versions to a small percentage of users first, watch metrics, then gradually roll out (or abort). This is how Netflix, Uber, and Google deploy safely.

---

## Task

### Part 1 — Why Progressive Delivery?
Standard rolling deployment: replaces pods one by one. If there's a bug, 50% of users see it before you notice.

Progressive delivery: route 5% of traffic to new version → check metrics → if good, 20% → 50% → 100%.

If error rate spikes at 5%: roll back immediately. Only 5% of users saw the bad version.

### Part 2 — Install Argo Rollouts
```bash
# Install Argo Rollouts
kubectl create namespace argo-rollouts
kubectl apply -n argo-rollouts \
    -f https://github.com/argoproj/argo-rollouts/releases/latest/download/install.yaml

# Install the kubectl plugin
brew install argoproj/tap/kubectl-argo-rollouts

# Verify
kubectl argo rollouts version
```

### Part 3 — Canary Rollout
```yaml
# rollout.yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 10
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:v1
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 100m
              memory: 128Mi

  strategy:
    canary:
      # Traffic routing via ingress (need nginx ingress controller)
      canaryService: myapp-canary
      stableService: myapp-stable
      trafficRouting:
        nginx:
          stableIngress: myapp-ingress

      steps:
        - setWeight: 5      # 5% to canary
        - pause: {duration: 2m}   # wait 2 minutes, watch metrics
        
        - analysis:
            templates:
              - templateName: success-rate
            args:
              - name: service-name
                value: myapp-canary
        
        - setWeight: 20     # 20% to canary
        - pause: {duration: 5m}
        
        - setWeight: 50
        - pause: {duration: 5m}
        
        - setWeight: 100    # full rollout
```

### Part 4 — Stable and Canary Services
```yaml
# stable-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-stable
spec:
  selector:
    app: myapp
  ports:
    - port: 8080

---
# canary-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-canary
spec:
  selector:
    app: myapp
  ports:
    - port: 8080
```

Argo Rollouts manages pod label changes to route traffic between stable and canary.

### Part 5 — Automated Analysis (Metrics-Based Promotion)
```yaml
# analysis-template.yaml
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
    - name: service-name
  metrics:
    - name: success-rate
      interval: 30s
      count: 5         # measure 5 times
      successCondition: result[0] >= 0.99   # 99% success rate
      failureLimit: 2  # fail if 2+ measurements fail
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service-name}}",status!~"5.."}[2m]))
            / sum(rate(http_requests_total{service="{{args.service-name}}"}[2m]))

    - name: p99-latency
      interval: 30s
      count: 5
      successCondition: result[0] < 1.0    # under 1 second
      failureLimit: 2
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            histogram_quantile(0.99, 
              rate(request_duration_seconds_bucket{service="{{args.service-name}}"}[2m])
            )
```

### Part 6 — Trigger and Watch a Canary Deployment
```bash
# Update the image (triggers rollout)
kubectl argo rollouts set image myapp myapp=myapp:v2

# Watch the rollout in terminal
kubectl argo rollouts get rollout myapp --watch

# Open the Argo Rollouts dashboard
kubectl argo rollouts dashboard

# The rollout will:
# 1. Start 5% canary
# 2. Run analysis
# 3. If metrics pass → increase to 20%
# 4. Continue until 100%

# Manually promote (skip pause)
kubectl argo rollouts promote myapp

# Abort (rollback to stable)
kubectl argo rollouts abort myapp
```

### Part 7 — Simulate a Bad Deployment
```bash
# Deploy a version with bugs (high error rate)
# Modify your app to return 500 for 20% of requests:
docker build --build-arg ERROR_RATE=0.2 -t myapp:v3-bad .
kubectl argo rollouts set image myapp myapp=myapp:v3-bad

# Watch the analysis fail after 5% traffic
kubectl argo rollouts get rollout myapp --watch

# When error rate > 1% in analysis:
# Rollout automatically aborts
# Traffic returns to 100% stable (v2)

# Verify rollback
kubectl argo rollouts get rollout myapp
# STABLE: myapp:v2
# CANARY: myapp:v3-bad (aborted)
```

### Part 8 — Blue-Green Deployment
```yaml
strategy:
  blueGreen:
    activeService: myapp-active       # production traffic
    previewService: myapp-preview     # test traffic (internal only)
    autoPromotionEnabled: false       # manual promotion
    scaleDownDelaySeconds: 30         # keep old version for 30s after switch
    prePromotionAnalysis:             # run analysis before promoting
      templates:
        - templateName: smoke-test
    postPromotionAnalysis:            # run analysis after promoting
      templates:
        - templateName: success-rate
```

```bash
# Trigger blue-green update
kubectl argo rollouts set image myapp myapp=myapp:v2

# Test the preview version (before promotion)
kubectl port-forward svc/myapp-preview 8081:8080
curl http://localhost:8081/health

# Manually promote after testing
kubectl argo rollouts promote myapp

# Both versions exist briefly, then old is scaled down
```
