### Part 1 — Git Internals
```bash
# Every Git object is content-addressed (SHA-1 hash)
# Look at the object database
git init demo-repo && cd demo-repo
echo "hello" > file.txt
git add file.txt
git commit -m "first commit"

# Examine the objects Git created
git log --oneline
git cat-file -p HEAD          # see the commit object
git cat-file -p HEAD^{tree}   # see the tree object
git cat-file -p <blob-sha>    # see the blob (file content)

# See the ref structure
cat .git/HEAD
cat .git/refs/heads/main
ls .git/objects/
```

Draw a diagram showing how commit → tree → blob objects link together.

### Part 2 — GitFlow Simulation
GitFlow uses: `main`, `develop`, `feature/*`, `release/*`, `hotfix/*`

```bash
# Set up GitFlow structure
git checkout -b develop
git push -u origin develop

# Feature branch
git checkout -b feature/user-auth develop
# ... make commits ...
git checkout develop
git merge --no-ff feature/user-auth  # always --no-ff to preserve branch history
git branch -d feature/user-auth

# Release branch
git checkout -b release/1.0.0 develop
# ... bump version, final fixes ...
git checkout main
git merge --no-ff release/1.0.0
git tag -a v1.0.0 -m "Release version 1.0.0"
git checkout develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0

# Hotfix
git checkout -b hotfix/fix-login-crash main
# ... fix ...
git checkout main && git merge --no-ff hotfix/fix-login-crash
git tag -a v1.0.1 -m "Hotfix 1.0.1"
git checkout develop && git merge --no-ff hotfix/fix-login-crash
git branch -d hotfix/fix-login-crash
```

Explain the difference between `merge --no-ff`, `merge --ff-only`, and `rebase`.

### Part 3 — Trunk-Based Development
Trunk-based development uses short-lived branches (< 1 day ideally) and feature flags:

```bash
# Short-lived feature branch
git checkout -b feat/add-search main
# ... make 1-3 commits ...
git fetch origin
git rebase origin/main  # keep up to date
git push origin feat/add-search
# Open PR → merge quickly → delete branch

# Interactive rebase to clean up commits before PR
git rebase -i HEAD~3  # squash/reword last 3 commits
```

When should you use GitFlow vs trunk-based? Write a 1-paragraph comparison.

### Part 4 — Merge Conflict Resolution
Create a conflict scenario:
```bash
git init conflict-demo && cd conflict-demo
echo "line 1\nline 2\nline 3" > file.txt
git add . && git commit -m "initial"

git checkout -b branch-a
echo "line 1\nMODIFIED BY A\nline 3" > file.txt
git commit -am "change from branch A"

git checkout main
echo "line 1\nMODIFIED BY MAIN\nline 3" > file.txt
git commit -am "change from main"

git merge branch-a   # this will conflict
```

Resolve the conflict. Then try:
```bash
# "Ours" strategy — take our version entirely
git merge -X ours branch-a

# "Theirs" strategy — take their version entirely
git merge -X theirs branch-a

# Visual merge tool
git mergetool
```

### Part 5 — Git Reflog and Recovery
```bash
# The reflog is your safety net — every HEAD change is recorded
git reflog

# Accidentally deleted a branch?
git branch -D important-branch
git reflog  # find the last commit on that branch
git checkout -b important-branch <sha>

# Accidentally reset --hard?
git reset --hard HEAD~3
git reflog  # find the commit you lost
git reset --hard <lost-sha>

# Stash work in progress
git stash push -m "WIP: half-done feature"
git stash list
git stash pop
git stash drop
```

Demonstrate recovering a "lost" commit using reflog.
