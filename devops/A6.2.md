### Part 1 — IAM Fundamentals
Key concepts:
- **User**: a person or service with long-term credentials
- **Group**: collection of users sharing policies
- **Role**: assumed by services, applications, or users (temporary credentials)
- **Policy**: JSON document defining permissions
- **Principal**: entity that makes a request (user, role, service)

```bash
# Never use root account for anything after initial setup
# Create an admin IAM user immediately:
aws iam create-user --user-name admin
aws iam attach-user-policy \
    --user-name admin \
    --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Enable MFA for the admin user (do this in console)
```

### Part 2 — Least Privilege: Custom Policies
Bad practice: give everyone `AdministratorAccess`.  
Good practice: give only what's needed.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "S3ReadOnly",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    },
    {
      "Sid": "EC2ReadOnly",
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyDeleteAnything",
      "Effect": "Deny",
      "Action": [
        "s3:DeleteObject",
        "ec2:TerminateInstances"
      ],
      "Resource": "*"
    }
  ]
}
```

Create this policy and attach it to a user:
```bash
aws iam create-policy \
    --policy-name MyAppReadOnly \
    --policy-document file://policy.json

aws iam attach-user-policy \
    --user-name developer \
    --policy-arn arn:aws:iam::ACCOUNT_ID:policy/MyAppReadOnly

# Test the policy using IAM policy simulator
aws iam simulate-principal-policy \
    --policy-source-arn arn:aws:iam::ACCOUNT_ID:user/developer \
    --action-names s3:GetObject \
    --resource-arns arn:aws:s3:::my-app-bucket/file.txt
```

### Part 3 — IAM Roles for EC2 (Instance Profiles)
EC2 instances should access AWS services via roles, never hard-coded credentials:

```bash
# Create role for EC2 to access S3
aws iam create-role \
    --role-name EC2-S3-Role \
    --assume-role-policy-document '{
      "Version": "2012-10-17",
      "Statement": [{
        "Effect": "Allow",
        "Principal": {"Service": "ec2.amazonaws.com"},
        "Action": "sts:AssumeRole"
      }]
    }'

aws iam attach-role-policy \
    --role-name EC2-S3-Role \
    --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# Create instance profile and attach role
aws iam create-instance-profile --instance-profile-name EC2-S3-Profile
aws iam add-role-to-instance-profile \
    --instance-profile-name EC2-S3-Profile \
    --role-name EC2-S3-Role

# Launch EC2 with the role
aws ec2 run-instances \
    --image-id ami-xxx \
    --instance-type t3.micro \
    --iam-instance-profile Name=EC2-S3-Profile \
    ...
```

On the EC2 instance, credentials come from the metadata service:
```bash
# On the EC2 instance:
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/EC2-S3-Role
# Returns temporary access key + secret + session token
# The AWS SDK and CLI pick this up automatically
aws s3 ls  # works without any configuration!
```

### Part 4 — Cross-Account Access
Scenario: Account A (dev) needs to access resources in Account B (prod):

```bash
# In Account B (prod): create a role that Account A can assume
aws iam create-role \
    --role-name CrossAccountS3Reader \
    --assume-role-policy-document '{
      "Version": "2012-10-17",
      "Statement": [{
        "Effect": "Allow",
        "Principal": {
          "AWS": "arn:aws:iam::ACCOUNT_A_ID:root"
        },
        "Action": "sts:AssumeRole",
        "Condition": {
          "StringEquals": {
            "sts:ExternalId": "unique-external-id-here"
          }
        }
      }]
    }'

# In Account A: assume the role
aws sts assume-role \
    --role-arn arn:aws:iam::ACCOUNT_B_ID:role/CrossAccountS3Reader \
    --role-session-name MySession \
    --external-id unique-external-id-here

# Configure CLI profile that auto-assumes the role
# ~/.aws/config:
[profile prod-access]
role_arn = arn:aws:iam::ACCOUNT_B_ID:role/CrossAccountS3Reader
source_profile = default
external_id = unique-external-id-here

# Use it
aws --profile prod-access s3 ls
```

### Part 5 — Permission Boundaries
Permission boundaries limit the maximum permissions a user or role can have:
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:*", "ec2:*"],
    "Resource": "*"
  }]
}
```

```bash
# Apply boundary when creating a role
aws iam create-role \
    --role-name LimitedRole \
    --assume-role-policy-document file://trust-policy.json \
    --permissions-boundary arn:aws:iam::ACCOUNT_ID:policy/MyBoundary
```

Even if `LimitedRole` has AdministratorAccess policy attached, it can only do S3 and EC2 actions.

### Part 6 — Audit and Find Overprivileged Users
```bash
# Generate access advisor: what services has this user actually used?
aws iam generate-service-last-accessed-details \
    --arn arn:aws:iam::ACCOUNT_ID:user/developer

# List users and their last activity
aws iam get-credential-report  # download CSV
aws iam generate-credential-report

# Find users with no MFA
aws iam list-users | jq '.Users[].UserName' | while read user; do
    mfa=$(aws iam list-mfa-devices --user-name "$user" | jq '.MFADevices | length')
    if [ "$mfa" -eq 0 ]; then echo "NO MFA: $user"; fi
done

# Find unused access keys (not used in 90 days)
aws iam list-users --query 'Users[*].UserName' --output text | tr '\t' '\n' | while read user; do
    aws iam list-access-keys --user-name "$user" \
        --query 'AccessKeyMetadata[?Status==`Active`].[UserName,AccessKeyId,CreateDate]' \
        --output table
done
```
