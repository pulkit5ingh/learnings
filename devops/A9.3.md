# A9.3 — Chaos Engineering: Game Day on a Staging Cluster

**Phase:** 9 — Advanced: GitOps, SRE & Platform Engineering  
**Topic:** Chaos engineering, resilience testing  
**Type:** Assignment

---

## Goal
Run a controlled chaos experiment to discover weaknesses in your system before production failures do. Chaos engineering is about building confidence in the system's behavior under turbulent conditions.

---

## Task

### Part 1 — Chaos Engineering Principles
The process:
1. **Define steady state**: what does "normal" look like? (error rate < 0.1%, p99 < 500ms)
2. **Hypothesize**: "The system will maintain steady state even when [experiment]"
3. **Introduce variables**: inject failures in a controlled way
4. **Observe**: does steady state hold?
5. **Learn**: if not, fix it. If yes, try harder experiments.

**Start small**: chaos on one service in staging, not all services in production.

### Part 3 — Experiment 1: Pod Kill
**Hypothesis**: "When a pod is killed, the service should recover in < 30 seconds and maintain availability > 99%"

```bash
# Before experiment: measure steady state
# Watch error rate and latency in Grafana
# Set up a load generator:
kubectl run load-gen --image=busybox -- sh -c \
    "while true; do wget -q -O- http://api-service:8080/health; sleep 0.1; done"

# Identify a pod to kill
kubectl get pods -l app=api

# Kill a pod
kubectl delete pod api-deploy-xxx --grace-period=0

# Watch recovery:
kubectl get pods -l app=api -w
# New pod should start within 10 seconds

# Check: did the service stay up during recovery?
# Check Grafana: any spike in error rate? (should be minimal with 3+ replicas)
```

```yaml
# LitmusChaos experiment definition
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: pod-kill-test
spec:
  appinfo:
    appns: default
    applabel: app=api
  chaosServiceAccount: pod-kill-sa
  experiments:
    - name: pod-delete
      spec:
        components:
          env:
            - name: TOTAL_CHAOS_DURATION
              value: "60"    # 60 seconds of chaos
            - name: CHAOS_INTERVAL
              value: "10"    # kill a pod every 10 seconds
            - name: FORCE
              value: "false" # graceful termination
```

### Part 4 — Experiment 2: Network Latency Injection
**Hypothesis**: "When database latency increases to 500ms, API P99 should stay below 2 seconds"

```bash
# Using tc (traffic control) to inject latency inside a container
kubectl exec -it api-pod-xxx -- sh

# Inside the container:
apt-get install -y iproute2
tc qdisc add dev eth0 root netem delay 500ms 100ms distribution normal
# Adds 500ms ± 100ms latency to all outgoing traffic

# Run your load test while this is active
# Observe Grafana latency panel

# Remove the chaos
tc qdisc del dev eth0 root netem
```

With LitmusChaos:
```yaml
experiments:
  - name: pod-network-latency
    spec:
      components:
        env:
          - name: NETWORK_INTERFACE
            value: eth0
          - name: NETWORK_LATENCY
            value: "2000"    # 2 second latency
          - name: TOTAL_CHAOS_DURATION
            value: "60"
          - name: TARGET_PODS
            value: "api-deploy-xxx"
```

### Part 5 — Experiment 3: Resource Stress
**Hypothesis**: "When a pod uses 80% CPU, HPA should scale within 3 minutes"

```bash
# Stress the CPU
kubectl exec -it api-pod-xxx -- sh
apt-get install -y stress-ng
stress-ng --cpu 2 --timeout 120s

# Watch HPA:
kubectl get hpa -w

# Watch pod count:
kubectl get pods -l app=api -w
```

### Part 6 — Experiment 4: Database Outage
**Hypothesis**: "When the database goes down, the API should return 503 (not 500), degrade gracefully, and recover within 60 seconds of database returning"

```bash
# Take down the database pod
kubectl scale statefulset postgres --replicas=0

# Observe:
# - Does the API return 503 with a clear error message?
# - Or does it return 500 (unhandled exception)?
# - Does it crash entirely?
# - When database comes back, does the API reconnect automatically?

# Restore
kubectl scale statefulset postgres --replicas=1
```

### Part 7 — Game Day Report
After running experiments, write a report:

```markdown
# Chaos Game Day Report — June 2025

## Service: API Service
## Environment: Staging

## Experiments Run

### Experiment 1: Pod Kill
- **Hypothesis**: Recovery < 30s, availability > 99%
- **Result**: PASS ✅
  - New pod started in 8 seconds
  - 3 requests failed (< 0.01% error rate)
  - Grafana showed brief spike then recovery

### Experiment 2: Network Latency (500ms to DB)
- **Hypothesis**: API P99 < 2s
- **Result**: FAIL ❌
  - API P99 jumped to 4.5 seconds
  - Root cause: 10 sequential database calls per request (N+1 pattern)
  - **Fix**: Batch the queries into a single call

### Experiment 3: CPU Stress
- **Hypothesis**: HPA scales within 3 minutes
- **Result**: PASS ✅
  - HPA scaled from 2 to 5 replicas in 2m 15s

### Experiment 4: Database Outage
- **Hypothesis**: 503 response, graceful degradation
- **Result**: PARTIAL ⚠️
  - App returned 503 correctly ✅
  - But reconnection took 45 seconds after DB returned ✅ (under 60s)
  - Log messages were cryptic — need better error messages ❌

## Action Items
1. [HIGH] Fix N+1 query pattern (found in Experiment 2)
2. [MEDIUM] Improve database reconnection error logging
3. [LOW] Add circuit breaker for database connections

## Next Game Day
- Test with pod memory pressure (OOMKill scenario)
- Test with partial network partition (some pods can't reach DB)
```
