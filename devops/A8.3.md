# A8.3 — HashiCorp Vault: Dynamic Secrets for Applications

**Phase:** 8 — DevSecOps & Compliance  
**Topic:** Secrets management, Vault  
**Type:** Assignment

---

## Goal
Use HashiCorp Vault to manage secrets properly. Hardcoding secrets in code or environment variables is a massive security risk. Vault provides dynamic, short-lived credentials with full audit trails.

---

## Task

### Part 1 — Run Vault in Dev Mode
```bash
# Install Vault
brew install vault

# Start Vault in dev mode (never do this in production)
vault server -dev -dev-root-token-id="root"
# Keep this running in another terminal

# In a new terminal:
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='root'

# Verify
vault status
```

Or run with Docker:
```yaml
# docker-compose.vault.yml
services:
  vault:
    image: hashicorp/vault:latest
    ports:
      - "8200:8200"
    environment:
      - VAULT_DEV_ROOT_TOKEN_ID=root
      - VAULT_DEV_LISTEN_ADDRESS=0.0.0.0:8200
    cap_add:
      - IPC_LOCK
```

### Part 2 — Basic Secret Operations
```bash
# Write a secret
vault kv put secret/myapp/database \
    host=db.internal \
    username=app_user \
    password=super-secret

# Read a secret
vault kv get secret/myapp/database
vault kv get -field=password secret/myapp/database

# List secrets
vault kv list secret/myapp

# Update a secret
vault kv patch secret/myapp/database password=new-password

# Delete
vault kv delete secret/myapp/database

# Get previous version
vault kv get -version=1 secret/myapp/database
```

### Part 3 — Dynamic Database Credentials (The Killer Feature)
Instead of a static password, Vault generates a new username/password for each request with a TTL:

```bash
# Enable the database secrets engine
vault secrets enable database

# Configure Postgres connection
vault write database/config/mydb \
    plugin_name=postgresql-database-plugin \
    connection_url="postgresql://{{username}}:{{password}}@localhost:5432/mydb?sslmode=disable" \
    allowed_roles="app-role" \
    username="vault_admin" \
    password="vault_admin_password"

# Create a role that defines what the temp user can do
vault write database/roles/app-role \
    db_name=mydb \
    creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
    revocation_statements="DROP ROLE IF EXISTS \"{{name}}\";" \
    default_ttl="1h" \
    max_ttl="24h"

# Request credentials
vault read database/creds/app-role
# Returns: username=v-root-app-role-xxx, password=yyy, lease_duration=1h

# After 1h, Vault automatically drops this user from PostgreSQL
```

```python
# In your application
import hvac

client = hvac.Client(url='http://vault:8200', token=os.getenv('VAULT_TOKEN'))

# Get dynamic database credentials
response = client.secrets.database.generate_credentials('app-role')
db_username = response['data']['username']
db_password = response['data']['password']

# Connect with these temp credentials
conn = psycopg2.connect(
    host='localhost',
    database='mydb',
    user=db_username,
    password=db_password
)
```

### Part 4 — Vault Policies
```hcl
# app-policy.hcl
# Allow reading static secrets for our app
path "secret/data/myapp/*" {
  capabilities = ["read", "list"]
}

# Allow generating database credentials
path "database/creds/app-role" {
  capabilities = ["read"]
}

# Allow renewing own token
path "auth/token/renew-self" {
  capabilities = ["update"]
}

# Deny everything else
path "*" {
  capabilities = ["deny"]
}
```

```bash
# Create the policy
vault policy write app-policy app-policy.hcl

# Create a token with this policy
vault token create -policy=app-policy -ttl=24h

# Test the policy (should work)
VAULT_TOKEN=$(vault token create -policy=app-policy -format=json | jq -r '.auth.client_token')
VAULT_TOKEN=$TOKEN vault kv get secret/myapp/database

# Test denial (should fail)
VAULT_TOKEN=$TOKEN vault kv put secret/other/secret key=val
```

### Part 5 — Vault Agent Sidecar (Auto-Auth)
In Kubernetes, a Vault Agent sidecar injects secrets directly into pod files:

```yaml
# Pod annotation to inject secrets
metadata:
  annotations:
    vault.hashicorp.com/agent-inject: "true"
    vault.hashicorp.com/role: "myapp"
    vault.hashicorp.com/agent-inject-secret-database: "secret/data/myapp/database"
    vault.hashicorp.com/agent-inject-template-database: |
      {{- with secret "secret/data/myapp/database" -}}
      export DB_HOST="{{ .Data.data.host }}"
      export DB_USERNAME="{{ .Data.data.username }}"
      export DB_PASSWORD="{{ .Data.data.password }}"
      {{- end }}
```

The secret is written to `/vault/secrets/database` inside the pod — no code changes needed.

### Part 6 — AWS Auth Method (For EC2 Instances)
EC2 instances can authenticate with Vault using their AWS identity:

```bash
# Enable AWS auth
vault auth enable aws

# Configure AWS
vault write auth/aws/config/client \
    access_key=AKIA... \
    secret_key=... \
    region=us-east-1

# Create a role: EC2 instances with this IAM role can authenticate
vault write auth/aws/role/myapp \
    auth_type=iam \
    bound_iam_principal_arn=arn:aws:iam::ACCOUNT:role/MyEC2Role \
    policies=app-policy \
    ttl=1h
```

On the EC2 instance:
```bash
# Authenticate using the instance's IAM role (no secrets needed!)
vault login -method=aws role=myapp
```
