# P8.2 — Secrets Management System with Vault + Kubernetes

**Phase:** 8 — DevSecOps & Compliance  
**Topic:** HashiCorp Vault, Kubernetes secrets  
**Type:** Project

---

## Goal
Build a zero-trust secrets management system where applications never see long-lived credentials — all secrets are dynamic, short-lived, and audited.

---

## What to Build
Deploy HashiCorp Vault in Kubernetes with:
- Dynamic database credentials (auto-rotating)
- PKI for TLS certificate issuance
- Kubernetes auth method (pods authenticate with ServiceAccount)
- Vault Agent sidecar injecting secrets
- Full audit logging
- Break-glass procedure for emergencies

## Architecture
```
Application Pod
    ├── App Container (never sees long-lived creds)
    └── Vault Agent Sidecar
            ↑ (reads secrets)
          Vault
            ├── Database Secrets Engine (PostgreSQL)
            ├── PKI Secrets Engine (TLS certs)
            ├── Kubernetes Auth Method
            └── Audit Logs → CloudWatch
```

## Deploy Vault in Kubernetes
```bash
# Install Vault with Helm
helm repo add hashicorp https://helm.releases.hashicorp.com

helm install vault hashicorp/vault \
    --namespace vault \
    --create-namespace \
    -f vault-values.yaml
```

```yaml
# vault-values.yaml
server:
  ha:
    enabled: true
    replicas: 3
    raft:
      enabled: true
  
  dataStorage:
    enabled: true
    size: 10Gi
    storageClass: gp2
  
  auditStorage:
    enabled: true
    size: 10Gi

  standalone:
    enabled: false

injector:
  enabled: true  # Vault Agent Injector
```

## Initialize and Configure Vault
```bash
# Initialize Vault (only done once)
kubectl exec -n vault vault-0 -- vault operator init \
    -key-shares=5 \
    -key-threshold=3 \
    -format=json > vault-init.json

# CRITICAL: Store the 5 unseal keys and root token securely
# Use AWS Secrets Manager or a hardware security module (HSM)
# NEVER commit vault-init.json to git

# Unseal Vault (need 3 of the 5 keys)
kubectl exec -n vault vault-0 -- vault operator unseal $(jq -r '.unseal_keys_b64[0]' vault-init.json)
kubectl exec -n vault vault-0 -- vault operator unseal $(jq -r '.unseal_keys_b64[1]' vault-init.json)
kubectl exec -n vault vault-0 -- vault operator unseal $(jq -r '.unseal_keys_b64[2]' vault-init.json)

# Login with root token (use only for initial setup)
export VAULT_TOKEN=$(jq -r '.root_token' vault-init.json)
```

## Configure Kubernetes Auth
```bash
# Enable Kubernetes auth method
vault auth enable kubernetes

# Configure with cluster's CA cert and API server URL
vault write auth/kubernetes/config \
    kubernetes_host="https://kubernetes.default.svc" \
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
    token_reviewer_jwt="$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)"

# Create a role: pods with this ServiceAccount can get DB credentials
vault write auth/kubernetes/role/myapp \
    bound_service_account_names=myapp-sa \
    bound_service_account_namespaces=production \
    policies=myapp-policy \
    ttl=1h
```

## Configure Dynamic Database Secrets
```bash
# Enable database engine
vault secrets enable database

# Configure PostgreSQL connection
vault write database/config/postgres \
    plugin_name=postgresql-database-plugin \
    connection_url="postgresql://{{username}}:{{password}}@postgres-service:5432/myapp" \
    allowed_roles="myapp-db-role" \
    username="vault_superuser" \
    password="initial_password"

# Create role for the app
vault write database/roles/myapp-db-role \
    db_name=postgres \
    creation_statements="
        CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';
        GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";
        GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO \"{{name}}\";
    " \
    revocation_statements="DROP ROLE IF EXISTS \"{{name}}\";" \
    default_ttl="1h" \
    max_ttl="24h"
```

## PKI for Internal TLS
```bash
# Enable PKI engine for internal CA
vault secrets enable pki
vault secrets tune -max-lease-ttl=87600h pki

# Generate root CA
vault write pki/root/generate/internal \
    common_name="myorg Internal CA" \
    ttl=87600h \
    key_type=ec \
    key_bits=256

# Create a role for issuing service certificates
vault write pki/roles/service-cert \
    allowed_domains="svc.cluster.local,myorg.internal" \
    allow_subdomains=true \
    max_ttl="720h"  # 30 days

# Issue a certificate for user-service
vault write pki/issue/service-cert \
    common_name="user-service.production.svc.cluster.local" \
    ttl="72h"
```

## Vault Agent Sidecar
```yaml
# Pod manifest with Vault Agent injection annotations
apiVersion: v1
kind: Pod
metadata:
  name: myapp
  namespace: production
  annotations:
    vault.hashicorp.com/agent-inject: "true"
    vault.hashicorp.com/role: "myapp"
    
    # Inject dynamic DB credentials
    vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/myapp-db-role"
    vault.hashicorp.com/agent-inject-template-db-creds: |
      {{- with secret "database/creds/myapp-db-role" -}}
      DATABASE_URL=postgresql://{{ .Data.username }}:{{ .Data.password }}@postgres:5432/myapp
      {{- end }}
    
    # Inject a TLS certificate
    vault.hashicorp.com/agent-inject-secret-tls: "pki/issue/service-cert"
    vault.hashicorp.com/agent-inject-template-tls: |
      {{- with secret "pki/issue/service-cert" "common_name=myapp.production.svc.cluster.local" -}}
      {{ .Data.certificate }}
      {{- end }}
    
    # Refresh credentials before they expire
    vault.hashicorp.com/agent-pre-populate-only: "false"
spec:
  serviceAccountName: myapp-sa
  containers:
    - name: myapp
      image: myapp:latest
      env:
        - name: VAULT_SECRETS_PATH
          value: /vault/secrets/
      # App reads /vault/secrets/db-creds for connection string
      # No credentials in environment variables or Kubernetes secrets
```

## Audit Configuration
```bash
# Enable audit logging to a file
vault audit enable file file_path=/vault/audit/audit.log

# Enable CloudWatch logging (via a Lambda that ships logs)
# Or use the syslog audit device
vault audit enable syslog tag=vault facility=AUTH

# Query audit logs to find who accessed what
cat /vault/audit/audit.log | jq '. | select(.type=="response") | {time, auth_type: .auth.display_name, path: .request.path}'
```

## Break-Glass Procedure
```markdown
# Vault Break-Glass Procedure

## When Vault is unavailable during an incident:
1. Use the break-glass credentials stored in AWS Secrets Manager
   `aws secretsmanager get-secret-value --secret-id vault/break-glass`

2. These are long-lived credentials for emergency use only

3. Record usage: log the incident ticket, your name, and time accessed

4. After incident: rotate break-glass credentials
   `vault write -f sys/generate-root/update ...`

5. Review audit log to verify break-glass usage was legitimate
```
