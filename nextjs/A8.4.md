# A8.4 — Environment Management

**Phase:** 8 — Deployment & CI/CD  
**Topic:** .env.local vs .env.production, NEXT_PUBLIC_ prefix, server-only env vars, runtime config  
**Type:** Assignment

---

## Goal
Implement a robust environment variable strategy with type validation, server-only enforcement, and correct scoping across environments.

## Background

Next.js loads environment variables from `.env` files in a specific order. Understanding this order is critical to avoid surprises:

1. `process.env` — OS environment variables (highest priority)
2. `.env.$(NODE_ENV).local` (e.g., `.env.development.local`) — per-machine overrides
3. `.env.local` — local overrides (not loaded in test environment)
4. `.env.$(NODE_ENV)` (e.g., `.env.production`) — environment-specific defaults
5. `.env` — base defaults

**NEXT_PUBLIC_ prefix**: Variables with this prefix are inlined into the JavaScript bundle at build time. They are accessible in the browser, Server Components, Route Handlers, and middleware. They CANNOT be changed after build — they're baked in.

**Server-only variables** (without the prefix) are never exposed to the browser. They're available in Server Components, Server Actions, Route Handlers, and middleware. Using them in Client Components causes an error at build time.

**server-only package**: Import this in files that should NEVER run in a browser. If a Client Component accidentally imports a file with `import "server-only"`, the build fails with a clear error message.

## Tasks

### 1. .env file hierarchy
```bash
# .env (committed — defaults for all environments)
NEXT_PUBLIC_APP_NAME="My App"
NEXT_PUBLIC_SUPPORT_EMAIL="support@myapp.com"

# .env.development (committed — development-specific)
NEXT_PUBLIC_APP_URL="http://localhost:3000"
NEXT_PUBLIC_DEBUG_MODE="true"

# .env.production (committed — production-specific)
NEXT_PUBLIC_APP_URL="https://myapp.com"
NEXT_PUBLIC_DEBUG_MODE="false"

# .env.local (NOT committed — local secrets and overrides)
DATABASE_URL="postgresql://postgres:password@localhost:5432/myapp_dev"
AUTH_SECRET="local-dev-secret-minimum-32-characters-here"
GOOGLE_CLIENT_ID="local-google-client-id"
GOOGLE_CLIENT_SECRET="local-google-client-secret"
```

### 2. Type-safe env validation with Zod
```ts
// src/env.ts
import { z } from "zod";

// --- Server-side env vars ---
const serverEnvSchema = z.object({
  NODE_ENV: z.enum(["development", "test", "production"]),
  DATABASE_URL: z.string().url(),
  AUTH_SECRET: z.string().min(32, "AUTH_SECRET must be at least 32 characters"),
  GOOGLE_CLIENT_ID: z.string().optional(),
  GOOGLE_CLIENT_SECRET: z.string().optional(),
});

// --- Client-side env vars (NEXT_PUBLIC_ prefix) ---
const clientEnvSchema = z.object({
  NEXT_PUBLIC_APP_URL: z.string().url(),
  NEXT_PUBLIC_APP_NAME: z.string(),
});

// Validate on startup. This throws if any required var is missing.
const serverEnv = serverEnvSchema.parse(process.env);
const clientEnv = clientEnvSchema.parse(process.env);

// Only export server vars if we're on the server
if (typeof window !== "undefined") {
  throw new Error("src/env.ts should only be imported from server-side code.");
}

export const env = {
  ...serverEnv,
  ...clientEnv,
};
```

pnpm add server-only
```

// src/lib/db.ts
import "server-only"; // ← If a Client Component imports this, the build fails

import { env } from "@/env";

// This file is safe to use DATABASE_URL because:
// 1. "server-only" prevents client imports
// 2. DATABASE_URL has no NEXT_PUBLIC_ prefix

export async function query(sql: string) {
  // connect using env.DATABASE_URL
  console.log("Querying with:", env.DATABASE_URL);
}
```

### 4. Client-safe env (public vars only)
```ts
// src/lib/client-env.ts
// This file CAN be imported by Client Components — it only exposes NEXT_PUBLIC_ vars

export const clientEnv = {
  appName: process.env.NEXT_PUBLIC_APP_NAME!,
  appUrl: process.env.NEXT_PUBLIC_APP_URL!,
  supportEmail: process.env.NEXT_PUBLIC_SUPPORT_EMAIL!,
};
```

### 5. Using env vars correctly in different contexts
```tsx
// Server Component — can use any env var
// src/app/dashboard/page.tsx
import { env } from "@/env";

export default async function DashboardPage() {
  // ✅ DATABASE_URL available here (server only)
  const user = await db.user.findFirst();
  return <div>{user?.email}</div>;
}
```
```tsx
// Client Component — only NEXT_PUBLIC_ vars
// src/components/Footer.tsx
"use client";

import { clientEnv } from "@/lib/client-env";

export function Footer() {
  return (
    <footer>
      <p>© {new Date().getFullYear()} {clientEnv.appName}</p>
      <a href={`mailto:${clientEnv.supportEmail}`}>Contact</a>
    </footer>
  );
}
```

### 6. Runtime environment detection
```ts
// src/lib/environment.ts
export const isProduction = process.env.NODE_ENV === "production";
export const isDevelopment = process.env.NODE_ENV === "development";
export const isTest = process.env.NODE_ENV === "test";
export const isServer = typeof window === "undefined";
export const isClient = !isServer;
export const isVercel = !!process.env.VERCEL;
export const vercelEnv = process.env.VERCEL_ENV; // "production" | "preview" | "development"
```

### 7. Accessing env vars at runtime (not build time)
```tsx
// IMPORTANT: NEXT_PUBLIC_ vars are STATIC — baked in at build time
// You cannot change them without rebuilding

// For truly dynamic runtime config, use a Route Handler:
// src/app/api/config/route.ts
export async function GET() {
  return Response.json({
    // Only expose safe, public configuration
    appName: process.env.NEXT_PUBLIC_APP_NAME,
    featureFlags: {
      newDashboard: process.env.FEATURE_NEW_DASHBOARD === "true",
      betaApi: process.env.FEATURE_BETA_API === "true",
    },
  });
}
```

### 8. Checking env vars in CI
```bash
# .github/workflows/ci.yml (snippet)
- name: Validate environment variables
  run: |
    if [ -z "$NEXT_PUBLIC_APP_URL" ]; then
      echo "❌ NEXT_PUBLIC_APP_URL is not set"
      exit 1
    fi
    echo "✅ Environment variables validated"
  env:
    NEXT_PUBLIC_APP_URL: ${{ vars.NEXT_PUBLIC_APP_URL }}
```
