## Goal
Protect routes at the middleware layer and implement role-based routing so admins and users see different dashboards.

## Background

Middleware in Next.js runs before every request and can redirect, rewrite, or modify responses. It runs on the Edge Runtime — meaning it is extremely fast but has access to only a subset of Node.js APIs. This makes it the ideal place for auth checks: protect entire route groups without adding auth checks to every page.

Auth.js v5 provides a `middleware` export that wraps your middleware function with auth state. You receive `auth` (the session) and `request` as arguments.

**Role-based access control (RBAC)** at the middleware level means you check `session.user.role` and redirect to appropriate pages. For example, `/admin` routes should only be accessible to users with `role === "admin"`. Other users get redirected to `/403` or `/dashboard`.

The `matcher` config in middleware restricts which paths run the middleware. You should always exclude static files (`_next/static`), public assets, and API auth routes from middleware.

## Tasks

### 1. Basic auth middleware
```ts
// src/middleware.ts
import { auth } from "@/auth";
import { NextResponse } from "next/server";

export default auth((req) => {
  const { nextUrl, auth: session } = req;
  const isLoggedIn = !!session;

  // Define protected path groups
  const isProtected = nextUrl.pathname.startsWith("/dashboard") ||
                      nextUrl.pathname.startsWith("/profile") ||
                      nextUrl.pathname.startsWith("/settings");
  const isAdminOnly = nextUrl.pathname.startsWith("/admin");
  const isAuthPage  = nextUrl.pathname.startsWith("/login") ||
                      nextUrl.pathname.startsWith("/register");

  // Redirect unauthenticated users away from protected routes
  if (isProtected && !isLoggedIn) {
    const loginUrl = new URL("/login", nextUrl.origin);
    loginUrl.searchParams.set("callbackUrl", nextUrl.pathname);
    return NextResponse.redirect(loginUrl);
  }

  // Redirect admin-only routes for non-admins
  if (isAdminOnly) {
    if (!isLoggedIn) {
      return NextResponse.redirect(new URL("/login", nextUrl.origin));
    }
    if ((session?.user as any)?.role !== "admin") {
      return NextResponse.redirect(new URL("/403", nextUrl.origin));
    }
  }

  // Redirect logged-in users away from auth pages
  if (isAuthPage && isLoggedIn) {
    return NextResponse.redirect(new URL("/dashboard", nextUrl.origin));
  }

  return NextResponse.next();
});

// Only run middleware on these paths — NOT on static files or API auth
export const config = {
  matcher: [
    "/((?!_next/static|_next/image|favicon.ico|public|api/auth).*)",
  ],
};
```

### 3. Role-based layouts
```tsx
// src/app/dashboard/layout.tsx
import { auth } from "@/auth";
import { redirect } from "next/navigation";
import { AdminNav } from "@/components/AdminNav";
import { UserNav } from "@/components/UserNav";

export default async function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  const session = await auth();
  if (!session) redirect("/login");

  const isAdmin = (session.user as any)?.role === "admin";

  return (
    <div className="flex min-h-screen">
      <aside className="w-56 border-r p-4">
        {isAdmin ? <AdminNav /> : <UserNav />}
      </aside>
      <main className="flex-1 p-6">{children}</main>
    </div>
  );
}
```

### 4. Admin section with additional server-side guard
```tsx
// src/app/admin/layout.tsx
import { auth } from "@/auth";
import { redirect } from "next/navigation";

// Double-check in layout (middleware is first line of defence)
export default async function AdminLayout({ children }: { children: React.ReactNode }) {
  const session = await auth();

  if (!session || (session.user as any)?.role !== "admin") {
    redirect("/403");
  }

  return (
    <div>
      <div className="bg-red-50 border-b border-red-200 px-6 py-2">
        <p className="text-sm text-red-700 font-medium">Admin Area</p>
      </div>
      {children}
    </div>
  );
}
```

### 5. Redirect to callbackUrl after login
```tsx
// src/app/login/page.tsx (handle callbackUrl)
import { auth, signIn } from "@/auth";
import { redirect } from "next/navigation";

export default async function LoginPage({
  searchParams,
}: {
  searchParams: { callbackUrl?: string };
}) {
  const session = await auth();
  if (session) redirect(searchParams.callbackUrl ?? "/dashboard");

  const callbackUrl = searchParams.callbackUrl ?? "/dashboard";

  return (
    <div className="min-h-screen flex items-center justify-center">
      <form action={async (formData: FormData) => {
        "use server";
        await signIn("credentials", {
          email: formData.get("email"),
          password: formData.get("password"),
          redirectTo: callbackUrl,
        });
      }} className="w-full max-w-sm space-y-4 p-6 border rounded-xl">
        <h1 className="text-2xl font-bold text-center">Sign in</h1>
        <input name="email" type="email" placeholder="Email"
          className="w-full px-3 py-2 border rounded" required />
        <input name="password" type="password" placeholder="Password"
          className="w-full px-3 py-2 border rounded" required />
        <input type="hidden" name="callbackUrl" value={callbackUrl} />
        <button type="submit" className="w-full py-2.5 bg-blue-600 text-white rounded">
          Sign in
        </button>
      </form>
    </div>
  );
}
```

## Deliverables
- [ ] Middleware redirects unauthenticated users from `/dashboard` to `/login?callbackUrl=/dashboard`
- [ ] After login, user is redirected back to the original URL
- [ ] `/admin/*` routes redirect non-admin users to `/403`
- [ ] Admin layout shows admin nav; user layout shows user nav
- [ ] Admin layout has a server-side guard (not just middleware)
- [ ] Login page redirects already-authenticated users away
- [ ] Static files and `/api/auth` are excluded from middleware matcher
